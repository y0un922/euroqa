# Eurocode QA 回答深度重构 — 设计文档

> Version: 1.0
> Date: 2026-04-10
> Status: Draft

## 1. 问题陈述

当前 Euro_QA 的回答已经具备基本的证据约束、引用展示和 `exact / open / exact_not_grounded` 三种生成模式，但在真实工程使用场景下仍存在一个关键缺口：

- 回答往往“有依据”，但“不够会讲”
- 对不熟悉 Eurocode 体系的中国工程师不够友好
- 用户能看到条文和结论，却不一定知道这条规则在工程上代表什么、适用边界是什么、项目中该如何落地

这会导致两个问题：

1. 对开放式问题，回答容易停留在摘要层，缺少教学式解释和工程动作
2. 对精确型问题，回答虽然更聚焦，但对“为什么这样规定、如何安全使用该结论”解释不够

与此同时，本系统服务的是落地工程项目，回答质量不能以“更长”替代“更稳”。因此本次改造的目标不是单纯增加篇幅，而是把回答重构为：

- 面向不懂欧标的中国工程师
- 解释更完整、更可落地
- 在证据不足时仍保持保守，不把未检索到的内容写成结论

## 2. 目标与非目标

### 2.1 本次目标

1. 保留现有三路生成模式：
   - `open`
   - `exact`
   - `exact_not_grounded`
2. 将三种模式统一重构为“保守教学式”回答
3. 让回答默认面向“不熟悉 Eurocode 的中国工程师”解释
4. 为不同模式定义不同的详细度边界，避免一刀切地扩写
5. 为提示词新增以下稳定约束：
   - 规则本意解释
   - 适用条件与边界
   - 工程使用动作
   - 常见误区或风险提示
   - 证据不足时的补查路径
6. 更新生成层测试，锁住“更详细”和“不能乱答”两个方向

### 2.2 非目标

- 不修改检索排序、rerank、Milvus 或 ES 基础设施
- 不改变前端回答渲染结构或 Evidence Panel UI
- 不新增新的回答模式
- 不把 `exact` 模式扩写成长篇培训讲义
- 不用工程经验替代规范证据

## 3. 用户画像与设计原则

### 3.1 目标用户

本次改造的默认目标用户是：

- 中国工程师
- 可能熟悉中国规范体系，但不熟悉 Eurocode 体系
- 需要把回答用于真实项目的设计、校核、审查或技术沟通

这意味着回答不能默认用户已理解以下内容：

- Eurocode 的章节组织方式
- 条款、表格、公式之间的引用关系
- National Annex 对结论边界的影响
- 欧洲规范术语在中国工程语境中的含义

### 3.2 设计原则

1. **证据优先**
   - 所有结论必须严格基于当前检索片段
   - 未被证据直接支持的内容不能写成规范要求

2. **教学化解释**
   - 不能只说“规范写了什么”
   - 必须解释“这条规则在说什么”

3. **工程落地**
   - 不能停留在条文翻译
   - 必须说明“在设计/校核/审查里怎么用”

4. **边界可见**
   - 不能把适用条件藏在细节里
   - 要主动指出适用边界、限制条件和需补查项

5. **模式分层**
   - `open` 允许充分展开
   - `exact` 保持“直接答案优先 + 中度解释”
   - `exact_not_grounded` 详细解释不确定性，但不替代证据下结论

## 4. 三种回答模式的重构方案

### 4.1 `open` 模式

#### 目标

用于规则解释、机理理解、参数含义、设计思路、综合性工程问答。回答应采用“工程培训手册式”风格，帮助用户系统理解规则，而不是只得到一个摘要。

#### 新结构

回答必须按以下结构输出：

- `### 先说结论`
- `### 这条规则在说什么`
- `### 适用条件与边界`
- `### 工程上怎么用`
- `### 容易出错的点`
- `### 当前依据`
- `### 还需要补充确认的内容`

#### 关键要求

- 第一部分先给出可直接确认的结论
- 第二部分必须把条文本意用中国工程师能理解的方式讲明白
- 第三部分必须说明适用对象、前提条件、可能影响结论的边界因素
- 第四部分必须转化为工程动作，例如：
  - 设计阶段要核对什么
  - 校核阶段要怎么使用参数或规则
  - 出图或审查时要重点关注什么
- 第五部分必须列出常见误读或高风险误用
- 若证据不完整，最后一节必须说明还缺什么，而不是默认补齐

### 4.2 `exact` 模式

#### 目标

用于条文定位、定义确认、公式或表格查值、精确性更强的问题。回答仍应保持“先直接回答”，但不能只给出一个孤立结论。

#### 新结构

- `### 直接答案`
- `### 关键依据`
- `### 这条规定应如何理解和使用`
- `### 使用时要再核对的条件`

#### 关键要求

- 第一节必须直接回答用户的问题
- 第二节必须给出最关键的条款、表格、公式或直接证据
- 第三节只做必要解释：
  - 说明这条规定在工程上意味着什么
  - 用户应如何使用这个结论
  - 必要时补一层术语解释
- 第四节列出真正影响结论成立的条件，不扩成开放式长文

#### 详细度边界

- 允许中度展开
- 不允许回到 8 段长模板
- 不允许写成完整培训讲义

### 4.3 `exact_not_grounded` 模式

#### 目标

用于用户提出精确问题，但当前证据还不足以直接得出明确结论的场景。回答必须让用户明白：

- 当前已经确认了什么
- 为什么还不能下结论
- 这种不确定性对项目意味着什么
- 下一步最应该补查什么

#### 新结构

- `### 当前能确认的内容`
- `### 为什么还不能直接下结论`
- `### 对工程决策的影响`
- `### 下一步应优先补查什么`

#### 关键要求

- 不再使用“过短 guardrail 模板”
- 要把“不确定性原因”讲清楚
- 要把“缺失证据会影响什么判断”讲清楚
- 要给出补查优先级
- 绝不允许把相关但不直接的材料包装成确定结论

## 5. 所有模式共享的硬约束

### 5.1 必须出现的解释元素

所有模式都必须在适当位置覆盖以下要素：

1. **术语解释**
   - 对中国工程师不直观的 Eurocode 术语，给出中文工程含义
   - 保留英文原词、条款号、表格号、公式号

2. **适用边界**
   - 明确指出适用于什么情况
   - 明确指出可能不适用于什么情况
   - 必要时指出是否可能受 National Annex 影响

3. **工程动作**
   - 告诉用户“接下来怎么用这条规则”
   - 不是只复述规范内容

4. **误区/风险**
   - 对高风险误读进行显式提醒

### 5.2 明确禁止的内容

所有模式均禁止：

- 把未检索到的内容写成 Eurocode 结论
- 用工程常识替代规范证据
- 为了显得详细而扩写空泛建议
- 在 `exact_not_grounded` 中暗示结论已经成立
- 在 `exact` 中扩写成大篇幅泛化说明

## 6. 生成层修改范围

本次改造仅修改生成层提示词与测试，不改 API 契约和前端展示。

### 6.1 主要修改点

- `server/core/generation.py`
  - 重构 `build_open_system_prompt()`
  - 重构 `build_exact_system_prompt()`
  - 重构 `build_exact_not_grounded_system_prompt()`
  - 如有必要，调整 `build_stream_system_prompt()` 的章节定义与说明文案
  - 保持 `_build_json_system_prompt()` 和 `_build_stream_mode_system_prompt()` 的模式映射关系不变

### 6.2 保持不变的内容

- `decide_generation_mode()` 的三路分流逻辑
- `generate_answer()` 与 `generate_answer_stream()` 的模式选择入口
- `build_prompt()` 的证据包组装逻辑
- 前端对 Markdown 的渲染方式

## 7. 测试策略

### 7.1 需要增强的测试方向

文件：

- `tests/server/test_generation.py`

至少补充以下断言：

1. `open` 模式
   - 断言新标题存在
   - 断言 prompt 中明确包含“中国工程师”“适用边界”“工程上怎么用”“容易出错的点”等要求
   - 断言仍然要求基于当前证据作答

2. `exact` 模式
   - 断言新结构为 4 节
   - 断言包含“直接答案”优先
   - 断言包含“如何理解和使用”的中度解释约束
   - 断言没有退回 8 段长模板

3. `exact_not_grounded` 模式
   - 断言新增“为什么还不能直接下结论”“对工程决策的影响”“下一步应优先补查什么”
   - 断言保留“不能把相关材料包装成直接依据”的 guardrail

4. 流式与非流式一致性
   - 断言 `generate_answer()` 和 `generate_answer_stream()` 仍依据相同模式选择不同 prompt
   - 断言路由逻辑不因文案重构而被破坏

### 7.2 验收标准

#### 阻断项

- 三类 prompt 已完成重构
- generation 相关测试通过
- 新测试能够同时覆盖“更详细”和“不能过度推断”
- `exact` 没被改成长篇泛讲义
- `exact_not_grounded` 没产生危险暗示性结论

#### 警告项

- 回答篇幅增加但解释要素未增加
- 有“工程动作”但没有“适用边界”
- 有“教学解释”但没有“证据约束”

## 8. 风险与回退策略

### 8.1 主要风险

1. `open` 模式变长但变空
2. `exact` 模式过度扩写，降低精确度
3. `exact_not_grounded` 变得更像“委婉给答案”

### 8.2 控制策略

- 用结构标题锁定回答骨架
- 用测试锁定关键 guardrail 语句
- 用模式分层避免“一套 prompt 走天下”

### 8.3 回退方式

若实测发现某类回答明显失控，可单独回退该模式的提示词，不影响其余两类模式。
