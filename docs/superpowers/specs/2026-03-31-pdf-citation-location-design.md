# Euro_QA PDF 引用定位与引用翻译解耦 — 设计文档

> Version: 1.0
> Date: 2026-03-31
> Status: Draft

## 1. 目标概述

### 1.1 背景

当前右侧 `EvidencePanel` 的设计是：

- 展示检索命中的 `original_text` chunk 文本
- 展示后端在问答阶段自动补齐的 `translation`
- 展示一张按页渲染的 PNG 预览图

这套设计已经能表达“来源是什么”，但还不能满足“直接在原始 PDF 中定位原文”的需求。用户点击引用后，看到的是 chunk 文本和页截图，而不是可交互的 PDF 原文定位视图。

同时，来源翻译当前和问答主链路绑定。也就是说，`/api/v1/query` 与 `/api/v1/query/stream` 会在返回 `sources` 时顺带补齐 `translation`。这让“是否需要引用翻译”无法由用户在界面上自行控制，也让主问答链路承担了额外的延迟和复杂性。

### 1.2 本次目标

本次改造要完成两个明确目标：

1. 将右侧证据面板从“chunk 文本 + 页截图”改为“PDF 定位查看器为主”
2. 将“引用翻译”从问答主链路中拆分出来，变为用户可控的独立能力

改造后的体验应满足以下要求：

- 主回答始终保持中文，不受本次改造影响
- 点击主回答中的引用后，右侧直接联动到对应 PDF 页
- 系统优先尝试在该页内搜索并高亮命中的引用文本
- 若高亮失败，仍停留在正确页，并清晰提示“已定位到页，但未能精确高亮”
- “引用翻译”是针对右侧当前引用原文的中文解释，而不是对主回答做翻译
- 用户可通过全局开关决定是否自动生成引用翻译
- 当全局开关从关闭切到开启时，当前正在查看的这条引用会立即补翻译

### 1.3 非目标

本次不包含以下范围：

- 修改主回答生成语言或回答结构
- 重新设计主工作区引用编号逻辑
- 为 PDF 提供坐标级框选定位
- 重做整套 PDF 解析/索引链路以产出 bbox 坐标
- 做跨会话、跨用户的翻译持久化存储
- 引入全文批量翻译或“翻译整轮全部来源”的能力

## 2. 设计原则

### 2.1 右侧面板以 PDF 原文为主，而不是以 chunk 文本为主

右侧面板的首要职责是让用户回到真实 PDF 原文，而不是继续阅读系统截取后的文本片段。因此：

- 中间主区域必须是 PDF 查看器
- `original_text` 不再作为主展示块常驻显示
- 顶部保留必要的元信息，底部保留引用翻译区

如果后续需要调试或排查，可再考虑折叠式原文文本辅助，但不作为本次主路径。

### 2.2 精确定位优先级低于稳定性

在当前数据结构中，`Source` 只有页码和文本内容，没有 PDF 内部坐标。因此本次定位策略采用：

- 先跳页
- 再用文本搜索尝试高亮
- 搜索失败则回退到仅跳页

系统不应为了追求“看起来更精确”而做过度激进的模糊匹配，以免把高亮放到错误位置，损害用户信任。

### 2.3 引用翻译必须从问答主链路中解耦

引用翻译是右侧面板的增强能力，不应继续绑定在 `/query` 或 `/query/stream` 的返回阶段。这样可以带来三个直接收益：

- 用户可以决定何时开启或关闭引用翻译
- 问答主链路不再为不需要的翻译付出额外成本
- 流式和非流式问答的行为保持更一致，来源翻译不再成为附带副作用

### 2.4 开关控制“是否生成”，而不是只控制“是否显示”

本次的“引用翻译”全局开关应该真正决定是否发起翻译请求，而不是让后端始终生成、前端仅做隐藏。否则：

- 无法真正降低请求成本
- 用户会误以为关闭后不会生成翻译，实际却仍在消耗资源
- 前后端状态语义不一致

因此，开关关闭时不应自动发起引用翻译请求。

## 3. 用户体验设计

### 3.1 右侧面板结构

右侧证据面板调整为三层结构：

1. 顶部：来源元信息 + “引用翻译”全局开关
2. 中部：PDF 查看器
3. 底部：引用翻译区

顶部保留的信息包括：

- 文档名
- 页码
- section
- clause
- 当前定位状态（已高亮 / 仅跳页 / 加载失败）

### 3.2 点击引用后的联动行为

当用户点击主回答中的某个引用时，右侧立即执行以下流程：

1. 激活该引用
2. 顶部来源信息更新
3. PDF 查看器切换到目标文档并跳到目标页
4. 用该引用的文本锚点在该页内执行搜索/高亮
5. 若“引用翻译”开关为开，则立即为该引用请求中文解释
6. 若“引用翻译”开关为关，则底部翻译区显示关闭状态

### 3.3 引用翻译全局开关行为

“引用翻译”全局开关控制的是右侧当前引用以及后续引用是否自动请求中文解释。

行为定义如下：

- 默认值为关闭
- 前端将该开关保存到本地会话存储；刷新页面后如果存在已保存值，则优先恢复用户上次选择
- 当开关从关闭切到开启时：
  - 当前激活引用立刻请求翻译
  - 后续点击的新引用也会自动请求翻译
- 当开关从开启切到关闭时：
  - 当前翻译区立即改为关闭状态
  - 后续点击的新引用不再自动请求翻译
  - 已经缓存的翻译结果可保留在内存中，但不自动显示

### 3.4 高亮失败时的退化体验

如果文本搜索无法在目标页内准确命中：

- PDF 查看器仍停留在正确页
- 顶部或 PDF 区显式提示“已定位到页，但未能精确高亮”
- 不自动退回 chunk 文本大段显示

这样既保留了“已经定位到真实原文页”的核心价值，又不会因为假命中造成误导。

## 4. 前端设计

### 4.1 `Source` 数据模型扩展

当前前端 `Source` 只包含：

- `file`
- `title`
- `section`
- `page`
- `clause`
- `original_text`
- `translation`

本次建议新增至少两个字段：

- `document_id`
  - 由后端直接提供，避免前端继续用 `matchSourceToDocumentId()` 做模糊匹配
- `locator_text`
  - 用于 PDF 页内搜索/高亮的文本锚点
  - 应比完整 `original_text` 更短、更适合命中

`translation` 字段仍然保留，但初始可为空字符串。

### 4.2 `useEuroQaDemo` 新增状态

前端工作台状态建议新增：

- `sourceTranslationEnabled: boolean`
- `sourceTranslationCache: Record<string, string>`
- `sourceTranslationLoadingKey: string | null`
- `pdfLocationStatus: "idle" | "highlighted" | "page_only" | "error"`

其中：

- `sourceTranslationEnabled` 控制是否自动请求引用翻译
- `sourceTranslationCache` 用于缓存本次会话中已获取的引用翻译
- `sourceTranslationLoadingKey` 用于避免切换引用时加载状态错乱
- `pdfLocationStatus` 用于驱动 UI 文案和状态标签

### 4.3 引用翻译缓存策略

本次不引入服务端持久化缓存，前端只做会话内缓存即可。

缓存键建议由以下字段组合得到：

- `document_id`
- `page`
- `locator_text`

要求：

- 同一会话中重复点击同一引用，不重复请求翻译
- 切换回答后，只要命中相同引用键，也可复用当前会话缓存
- 刷新页面后不要求保留该缓存，重新进入页面后按需重新请求即可

### 4.4 右侧 PDF 查看组件

右侧不再继续使用 `buildDocumentPreviewUrl()` 生成 PNG 页图作为主显示能力，而是引入一个真正的 PDF 查看组件。

组件职责建议拆分为：

- `EvidencePanel`
  - 负责整体布局、元信息、状态提示、引用翻译区
- `PdfEvidenceViewer`
  - 负责加载 PDF、跳页、搜索和高亮

`PdfEvidenceViewer` 的输入应至少包括：

- `documentId`
- `page`
- `locatorText`
- `onLocationResolved(status)`

### 4.5 顶部开关挂载位置

“引用翻译”全局开关不应和 `LLM 设置` 面板混在一起，因为两者语义不同：

- `LLM 设置` 是运行时模型配置
- “引用翻译”是右侧证据面板行为控制

建议将该开关放在右侧面板头部，而不是 `TopBar`。这样有两个好处：

- 用户在查看引用时能直接理解该开关只影响右侧引用翻译
- 不会让顶部栏承担过多与主工作区无关的全局设置

## 5. 后端设计

### 5.1 `Source` 构建逻辑调整

`server/core/generation.py` 中 `_build_sources_from_chunks()` 目前会直接把：

- `file`
- `title`
- `section`
- `page`
- `clause`
- `original_text`
- `translation=""`

组装为 `Source`。

本次需要扩展为同时补齐：

- `document_id`
- `locator_text`

建议原则：

- `document_id` 应直接由 `meta.source` 与当前文档体系映射得到，不再把映射工作留给前端
- `locator_text` 应来源于 `chunk.content` 的可搜索片段，长度需要受控，避免整段过长影响 PDF 页内命中

### 5.2 问答主链路不再自动补引用翻译

当前 `generate_answer()` 与 `generate_answer_stream()` 在构造 sources 后会调用：

- `_fill_missing_source_translations()`

本次应将这一步从默认路径中移除，改为：

- `query` / `queryStream` 返回的 `sources[].translation` 默认为空字符串
- 是否生成翻译由新接口单独负责

这样改动后：

- 主问答响应更聚焦
- 流式 `done` 事件与非流式 `QueryResponse` 行为保持一致
- 引用翻译只在用户明确需要时发生

### 5.3 PDF 文件接口

当前后端只提供：

- `GET /api/v1/documents`
- `GET /api/v1/documents/{doc_id}/page/{page}`

本次需要新增一个原始 PDF 文件接口，例如：

- `GET /api/v1/documents/{doc_id}/file`

职责：

- 返回 PDF 原文件内容
- 供右侧 PDF 查看器加载

页截图接口可保留，用于未来缩略图或兼容回退，但不再是主路径。

### 5.4 引用翻译接口

新增一个独立接口，例如：

- `POST /api/v1/sources/translate`

输入建议最少包含：

- `document_id`
- `file`
- `section`
- `clause`
- `page`
- `original_text`
- `locator_text`

输出建议为：

```json
{
  "translation": "设计使用年限应根据结构用途、维护条件以及预期服役场景加以规定。"
}
```

约束：

- 严格基于当前引用原文生成解释
- 不修改主回答内容
- 不返回额外的来源集合

### 5.5 引用翻译复用现有翻译能力

虽然问答主链路不再默认补翻译，但现有 `_build_source_translation_prompt()`、`_call_source_translation_llm()`、`_translate_source_batch()` 这些能力应尽量复用。

本次更适合做的是：

- 抽出一个面向单条/多条 source 的可复用服务函数
- 由 `/api/v1/sources/translate` 调用它

而不是保留“问答时顺手翻译”的路径。

## 6. PDF 定位与高亮策略

### 6.1 查看器技术路线

为了获得稳定的页跳转和文本搜索/高亮能力，右侧 PDF 查看器必须基于 PDF.js 生态实现，而不是浏览器原生 PDF 嵌入。

实现层可选择：

- 直接接入 `pdfjs-dist` 能力
- 或使用一个足够薄的 React 封装

但无论具体选型如何，都必须满足：

- 稳定页跳转
- 页内文本搜索
- 搜索命中后可视化高亮
- 失败时可回退到“只跳页”

### 6.2 文本锚点策略

`locator_text` 的目的不是完整展示原文，而是给页内搜索提供一个更容易命中的文本锚点。

建议规则：

- 优先截取 `original_text` 中前部连续、语义完整、可区分性高的文本
- 去除极端冗长、跨段、表格噪声过多的部分
- 保证搜索文本不会短到过度模糊，也不会长到完全无法命中

本次不要求做到“最佳锚点算法”，只要求比直接拿整段 chunk 搜索更稳。

### 6.3 定位状态机

右侧 PDF viewer 的定位状态建议固定为以下几类：

- `idle`
  - 尚未开始定位
- `highlighted`
  - 已跳页并成功高亮
- `page_only`
  - 已跳页，但未精确高亮
- `error`
  - PDF 加载或定位失败

UI 仅对用户暴露清晰语义：

- 已定位并高亮
- 已定位到页，但未能精确高亮
- 文档加载失败

### 6.4 不做激进模糊匹配

本次明确禁止以下行为：

- 未命中时自动在前后页扩散搜索
- 对短文本做高度模糊匹配后直接高亮
- 在用户看不见的情况下把“低置信匹配”伪装成成功高亮

如果当前页命中失败，就停留在该页并给出明确提示。这是比“可能高亮错地方”更安全的退化策略。

## 7. 兼容性与风险控制

### 7.1 对现有前端的兼容影响

受影响的主要前端区域包括：

- `EvidencePanel`
- `useEuroQaDemo`
- `lib/types.ts`
- `lib/api.ts`

主回答区的引用点击机制可保持不变，仍然通过 `activeReferenceId` 驱动右侧联动。

### 7.2 对现有后端的兼容影响

受影响的主要后端区域包括：

- `server/models/schemas.py`
- `server/core/generation.py`
- `server/api/v1/documents.py`
- 新增 `server/api/v1/sources.py` 或同等职责文件
- `server/api/v1/router.py`

老客户端如果仍消费 `sources[].translation`，在本次改造后会得到空字符串，但不会引发结构性错误。

### 7.3 主要风险

本次最大的技术风险不在翻译接口，而在 PDF 页内文本命中率。

具体风险点：

- PDF.js 文本层和原始 chunk 文本可能存在空白、连字符、换行差异
- `original_text` 直接拿来搜索可能过长或格式噪声过多
- 某些表格/公式型内容天然不适合文本高亮

对应控制策略：

- 明确引入 `locator_text`
- 接受“跳页成功但高亮失败”的可控退化
- 不把“高亮成功”作为唯一可接受结果

## 8. 验收标准

以下场景全部通过，才视为本次设计目标达成：

### 8.1 PDF 定位

- 点击任一回答引用后，右侧能打开对应 PDF 文档
- 右侧能跳转到引用对应页码
- 命中成功时，用户肉眼能看到明确高亮
- 命中失败时，仍然停留在正确页，并显示失败提示

### 8.2 引用翻译

- “引用翻译”开关关闭时，不自动发起翻译请求
- 开关从关闭切到开启时，当前引用立即发起翻译请求
- 开关开启后，后续点击的引用自动触发翻译
- 已翻译过的引用在同一会话中可直接复用结果
- 翻译失败时，仅影响右侧翻译区，不影响 PDF viewer

### 8.3 主问答链路

- 主回答始终保持中文输出
- `query` 与 `queryStream` 在默认路径下不再自动补来源翻译
- 去掉默认来源翻译后，不影响主回答和来源基础信息返回

### 8.4 退化与边界

- PDF 文件加载失败时，右侧显示错误状态但不导致页面崩溃
- 用户在翻译请求未完成前切换引用，不会出现旧请求覆盖新引用结果
- 关闭开关后，不再自动触发新引用翻译

---

## 9. 实施建议

本次改造建议按以下顺序推进：

1. 先扩展 `Source` 数据模型，补齐 `document_id` 与 `locator_text`
2. 再新增 PDF 文件接口与前端 PDF viewer
3. 然后移除问答主链路中的默认来源翻译
4. 最后新增独立引用翻译接口与前端全局开关、缓存逻辑

这样可以先把“PDF 定位主路径”建立起来，再切换翻译逻辑，降低联调复杂度。

---

## 10. 设计结论

本次改造采用以下最终方案：

- 右侧 `EvidencePanel` 改为以 PDF 查看器为主
- 主路径使用“跳页 + 当前页文本高亮”
- 高亮失败时退化为“仅跳页 + 明确提示”
- `Source.translation` 不再由问答主链路默认补齐
- 引用翻译改为用户可控的独立能力
- “引用翻译”全局开关放在右侧面板头部，而不是顶部栏

该方案能在不重构整套 PDF 解析体系的前提下，先满足“真实 PDF 定位”和“引用翻译用户可控”这两个最核心目标，并为后续升级到更精确的坐标级定位保留空间。
