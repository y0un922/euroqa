# 检索优化项目 — 风险评估

## 变更风险矩阵

### Phase A: 清理意图分类

| 变更 | 风险 | 等级 | 缓解 |
|------|------|------|------|
| 删除 IntentType 枚举 | 其他代码引用 | 低 | grep 确认只有已知调用方 |
| 删除 classify_intent | 精确引用查询可能召回下降 | 低 | rerank 会处理；且 extract_filters 仍在工作 |
| 简化 _merge_results | 改变合并顺序 | 低 | rerank 后顺序被重新排列 |
| 删除测试用例 | 测试覆盖下降 | 低 | 删除的都是 intent 相关测试 |

**总体评估**：低风险，因为 intent 分类对最终结果影响极小。

### Phase B: 检索参数调优

| 变更 | 风险 | 等级 | 缓解 |
|------|------|------|------|
| 增大 top_k | ES/Milvus 延迟增加 | 低 | 20→30 幅度不大 |
| 增大 rerank_top_n | prompt token 增加 | 中 | 需同步调整 token 预算 |
| 放宽 cross_doc_aggregate | 结果多样性下降 | 中 | 按场景动态调整 |

### Phase C: 召回优化

| 变更 | 风险 | 等级 | 缓解 |
|------|------|------|------|
| BM25 增加原始中文查询 | 中文在英文文档中匹配少 | 低 | embedding_text 可能含中文摘要 |
| rerank 使用原始问题 | 语义不匹配 | 中 | 可同时传原始+改写，或只用原始 |
| query expansion | LLM 调用增加延迟 | 中 | 可并行化 |

## 关键观察

### 为什么 intent 分类没用

1. **范围太窄**：只有 3 个类别，且只有 EXACT 有差异化处理
2. **正则不可靠**：`6.3.5` 会匹配 EXACT，但 "6.3.5条怎么理解" 其实需要概念解释
3. **rerank 抹平差异**：无论合并顺序如何，rerank 重新排序后原始顺序失效
4. **没有差异化检索策略**：不同 intent 用相同的 top_k、相同的 rerank 参数

### 当前召回瓶颈分析

1. **查询改写质量**：中文→英文关键词改写是单次 LLM 调用，可能遗漏关键语义
2. **BM25 搜索范围**：只用英文改写后的关键词搜 BM25，丢失了中文 embedding_text 的匹配机会
3. **候选池上限**：20+20 去重后约 25-30 个，可能不够
4. **cross_doc_aggregate**：max_per_source=3 在深度单文档查询中过于激进
5. **rerank 只用改写后的查询**：reranker 看不到用户原始意图

### 回答质量瓶颈

1. **token 预算严格**：parent_chunks 只在 child_tokens < 3000 时才包含
2. **source 翻译延迟**：每次回答都要额外 LLM 调用翻译 sources
3. **prompt 结构合理**：system prompt 质量不错，优化空间有限
