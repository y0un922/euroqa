# 风险评估

本文件聚焦“给 LLM 回复增加复制按钮，并支持整会话 Markdown 导出，内容含回答原文与本轮检索召回的全部上下文”的实施风险。

## 总结

总体风险评级：`中高`

原因不是界面复杂，而是“数据准确性优先”这一约束要求我们不能用现有 `sources` 去近似代替“全部召回上下文”。当前完整检索上下文只存在于后端请求生命周期中，尚未进入 API、前端状态或本地持久化层。

## 关键风险

### 1. API 当前无法表达“全部召回上下文”

- 等级：高
- 现状：
  - `server/core/retrieval.py` 返回 `chunks` 与 `parent_chunks`
  - `server/core/generation.py` 在 prompt 中使用它们
  - 但 `QueryResponse` 和流式 `done` 事件只返回 `sources`
- 风险：
  - 如果前端只拿 `sources` 做复制，会丢失未进入最终引用列表的其他召回片段
  - 与“复制本轮检索召回的全部上下文”的目标不一致
- 建议：
  - 增加显式的 `retrieval_context` 或 `retrieval_snapshot` 响应字段
  - 流式与非流式统一返回同一结构

### 2. 流式与非流式路径需要严格保持一致

- 等级：高
- 现状：
  - 流式路径：`queryStream()` + SSE `done`
  - 非流式路径：`query()`
  - 当前两条路径都能得到 `answer/sources/confidence`，但没有上下文快照字段
- 风险：
  - 若只给其中一条链路补字段，会导致：
    - 流式成功时可复制完整上下文
    - fallback 时只能复制部分信息
- 建议：
  - 所有新增字段都以 schema 为中心定义
  - 先补后端统一构造函数，再分别接到 `/query` 与 `/query/stream`

### 3. 整会话导出依赖前端本地状态完整性

- 等级：高
- 现状：
  - `ConversationManager` 只在后端保存最近几轮 `question/answer`
  - 浏览器 localStorage 才持久化完整 `messages`
- 风险：
  - 如果整会话导出依赖服务端重建，将拿不到每轮详细 `sources` 和未来新增的完整检索上下文
  - 如果依赖前端 `messages`，则必须保证新增字段也进入 `session.ts`
- 建议：
  - 将“导出整个会话”定义为“导出当前前端会话快照”
  - 在 `ChatTurn` 中直接存放导出所需的完整信息

### 4. 历史消息与新消息的数据不兼容

- 等级：中
- 现状：
  - 旧 localStorage 会话中没有未来新增字段
  - `session.ts` 已存在兼容恢复逻辑
- 风险：
  - 用户刷新后恢复旧消息时，单条复制可能缺字段
  - 整会话导出中不同消息的数据密度不一致
- 建议：
  - 为新字段提供兼容默认值
  - 导出逻辑对“上下文缺失”给出明确文本标记，而不是静默伪造

### 5. “回复原文”与“渲染后文本”不可混用

- 等级：中
- 现状：
  - `MainWorkspace.tsx` 先 `extractDisplayAnswer()`，再 `linkifyReferenceCitations()`，最后交给 `ReactMarkdown`
- 风险：
  - 如果复制逻辑从 DOM 或渲染后字符串取值，会把内部引用改写、UI 处理结果和原始 Markdown 混在一起
- 建议：
  - 单条复制必须基于 `ChatTurn.answer` 原始值
  - 引用来源与检索上下文作为额外 Markdown 节拼装

### 6. “全部召回上下文”的边界需要固定

- 等级：中
- 现状：
  - 后端实际拥有两个层次：
    - `chunks`：最终用于回答的主检索结果
    - `parent_chunks`：按 token budget 截断后加入 prompt 的章节级扩展上下文
- 风险：
  - 如果不先定义导出边界，容易出现实现歧义：
    - 只导出 `chunks`
    - 导出 `chunks + parent_chunks`
    - 导出未重排候选集合
- 建议：
  - 在 Phase 2 里明确“全部召回上下文”的协议定义
  - 从当前代码看，最合理的语义是“实际传给生成层的上下文”，即：
    - 最终 `chunks`
    - 被 `build_prompt()` 真正纳入 prompt 的 `parent_chunks`

### 7. UI 入口位置不当会影响工作台可读性

- 等级：中
- 现状：
  - 主回答区信息密度已经较高，有回答状态、reasoning、引用来源
  - 顶栏与侧栏则更适合会话级操作
- 风险：
  - 把单条复制和整会话导出都塞到消息头，容易让主内容区过于拥挤
- 建议：
  - 单条复制放在消息级 header 或回答卡片右上角
  - 整会话导出放在 `TopBar` 或 `Sidebar`

### 8. 脏工作区增加并行改动冲突概率

- 等级：中
- 现状：
  - 当前仓库有大量未提交改动和未跟踪文件
- 风险：
  - 修改问答链路、前端状态和测试时容易与现有本地工作重叠
- 建议：
  - 本轮严格限定在 feature 相关文件内增量修改
  - 实施前先在计划中明确文件边界

## 已存在的可复用实现模式

以下已有模式能降低实施风险：

- `reasoning` 的跨层扩展模式
  - 已证明“后端 SSE 新事件 + 前端类型 + session 持久化 + UI 展示”这条链路可行
- `source.translation` / `highlight_text` 的 schema 扩展模式
  - 已证明前后端响应字段可以通过类型与测试一起增量演进
- `session.ts` 的兼容恢复逻辑
  - 可用于为新增字段提供迁移默认值

结论：这不是一次从零设计的能力，已有多次“响应字段扩展 + 前端状态同步”的成功路径可以复用。

## 推荐风险应对策略

### 协议层

- 先定义新的检索上下文快照 schema，再写 UI
- 让 `/query` 与 `/query/stream` 共享同一份快照构造逻辑

### 状态层

- 把复制/导出所需数据直接挂到 `ChatTurn`
- `session.ts` 同步支持序列化和恢复

### 展示层

- 单条复制与整会话导出共用同一个 Markdown 序列化函数
- 不从渲染后的 HTML 读取内容

### 验证层

- 后端测试覆盖：
  - 新字段 schema
  - 流式/非流式一致性
- 前端测试覆盖：
  - SSE 新字段解析
  - session 恢复
  - Markdown 导出格式

## Phase 2 拆解时必须回答的问题

下一阶段任务拆解前，需要固定这 4 个设计判断：

1. “全部召回上下文”最终定义为哪些后端对象
2. 新字段是直接挂在 `QueryResponse` 顶层，还是挂在每条 `ChatTurn` 的专用结构中
3. 整会话导出入口放在 `TopBar` 还是 `Sidebar`
4. 历史旧消息在缺失上下文时，导出文案如何降级提示
