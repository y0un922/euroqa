# 欧洲规范问答系统技术路线说明

## 1. 系统是做什么的，解决了什么问题

这个系统本质上是一个面向工程规范场景的智能问答工具。

它面向的业务问题非常明确：中国工程师在欧洲参与工程设计和施工时，经常需要查阅欧洲规范，但实际工作中会遇到几个典型障碍：

- 工程师习惯用中文提问和理解问题。
- 规范原文通常是英文，且结构复杂、章节繁多。
- 单靠人工查找条款效率低，容易漏条款、错理解。
- 普通聊天机器人虽然能回答问题，但往往缺少明确依据，难以用于工程判断。

这个系统的目标，就是把“中文提问”“欧洲规范检索”“中文解释回答”“原文条款举证”连接成一个闭环。用户提出问题后，系统不仅给出答案，还会展示对应的原文规范条目、页码和定位信息，帮助用户核对依据。

从业务价值看，它解决的是“跨语言、跨规范体系的工程查标效率和可靠性问题”，而不是一个泛化的聊天机器人问题。

## 2. 整个系统的总体思路是什么

这套系统采用的是“检索增强生成”路线，也就是先检索，再回答，而不是让大模型直接凭训练记忆作答。

它的整体思路可以概括为四句话：

1. 先把本地 PDF 规范文档离线处理成结构化、可检索的知识库。
2. 用户提问时，先理解问题，再到知识库中找最相关的条款和片段。
3. 把检索到的内容作为依据交给大模型生成中文回答。
4. 在返回答案时，同时把对应原文依据和定位信息展示出来。

这条路线的关键，不是单纯“接一个大模型”，而是把两个能力融合起来：

- 一方面，系统需要像搜索引擎一样，在复杂规范文档中准确找到相关条款。
- 另一方面，系统需要像专业助理一样，把找到的条款转化成工程师容易理解的中文解释。

从实现方式看，这不是“文档搜索”和“AI 回答”各自独立，而是前后串联：

- 检索负责找依据。
- 大模型负责组织答案、解释上下文和输出中文说明。
- 证据展示模块负责把回答重新绑定到原始规范内容上。

因此，这是一条“以规范检索为基础、以大模型生成增强表达、以原文举证保障可靠性”的技术路线。

## 3. 系统主要包含哪些模块

从整体架构看，系统主要可以分为以下几个模块。

### 3.1 用户交互模块

负责接收用户中文提问、管理会话状态、展示回答结果，并把引用条款、PDF 原文页和高亮定位一起呈现出来。

这一层的重点不是单纯聊天界面，而是“回答区 + 证据区”的双区协同，让用户能同时看到解释和依据。

### 3.2 文档处理模块

负责把本地欧洲规范 PDF 做离线解析，抽取正文、章节结构、表格、公式、页码、位置等信息，形成后续检索可用的数据基础。

这一步决定了系统能不能从“PDF 文件”走向“结构化知识”。对于工程规范来说，这一步尤其重要，因为规范并不是简单连续文本，而是包含章节编号、表格、公式、图示等复杂内容。

### 3.3 文档结构化与分块模块

负责把规范内容拆分成适合检索的知识单元。

系统并不是把整本规范原样丢给向量库，而是会按章节、段落和特殊元素进行分层切分，形成适合召回的文本块，并保留这些块与上级章节、页码、条款编号之间的关系。

这样做的目的，是让系统既能命中局部条款，又能在需要时补充更完整的上下文。

### 3.4 问题理解模块

负责理解用户输入的中文问题，包括：

- 清洗无关内容和潜在干扰内容。
- 抽取可能的文档过滤条件。
- 识别条款、表格、公式等显式线索。
- 把中文问题扩展成更适合检索英文规范的查询表达。

这一模块对本项目特别关键，因为用户说中文，而规范原文大多是英文。系统如果不先做问题理解和中英术语对齐，检索效果会明显下降。

### 3.5 内容检索模块

负责从知识库中找出与问题最相关的规范内容。

它不是只用一种检索方式，而是结合了语义检索和关键词检索两种能力，再进行去重、重排和上下文补充。这样既能支持概念性问题，也能兼顾条款编号、表格编号、公式编号这类精确查找需求。

### 3.6 模型生成模块

负责基于检索到的规范片段生成中文回答。

这一模块并不是“自由聊天”，而是把问题、检索结果、最近会话上下文、术语信息等一起组织成提示词，让模型围绕已有依据作答。也就是说，模型的作用更接近“基于依据进行解释和归纳”，而不是脱离文档独立判断。

### 3.7 原文依据展示模块

负责把命中的条款、页码、片段和 PDF 页面定位展示给用户。

这也是系统区别于普通问答机器人的重要模块。工程场景里，答案只是第一层，证据才是最终能支撑设计和施工判断的关键。

## 4. 用户提问后，系统内部的处理流程是什么

从一次完整问答来看，系统内部大致经历以下步骤。

### 第一步：接收问题并建立本轮会话

用户在前端输入中文问题后，请求会被送到后端问答接口。系统会结合当前会话上下文，准备本轮处理所需的参数和历史信息。

### 第二步：对问题做理解和清洗

系统先不急着检索，而是先分析问题内容，包括：

- 问题本身在问什么。
- 是否指定了某本规范或某类条款。
- 是否包含表格、公式、章节编号等显式线索。
- 是否存在不必要或不安全的提示性内容。

随后，系统会把中文问题扩展为更适合检索规范原文的查询表达。根据代码实现，这一步通常会把问题转成多条英文检索表达，并结合术语表做中英映射。

### 第三步：执行混合检索

系统会同时走两类检索路径：

- 一类是向量检索，用于找语义相关内容。
- 一类是关键词检索，用于找明确词项、编号和固定表达。

两路结果出来后，系统会进行去重和合并，并进一步做重排，尽量把最相关的规范内容排到前面。

根据代码实现，系统还会用原始中文问题再补做一轮召回和重排。这说明系统并不是完全依赖“翻成英文后再检索”，而是在中文表达和英文规范之间做了双重保险。

### 第四步：补充上下文

工程规范的问题，往往不能只看一句命中的条文，还要看它所在的小节或上下文说明。

因此，系统在命中具体片段后，还会进一步补充其上级章节或父级内容，让后续生成阶段获得更完整的语义背景，减少“只看局部句子就误解条文”的风险。

### 第五步：组织提示并调用大模型生成答案

在拿到检索结果后，系统会把以下内容一起送入大模型：

- 用户当前问题。
- 检索到的相关条款片段。
- 更完整的章节上下文。
- 必要的术语信息。
- 最近几轮会话内容。

模型基于这些内容生成中文回答，重点输出解释、归纳和适度的工程化表达。

### 第六步：构造原文依据和定位信息

答案生成后，系统不会只返回一段文本，而是同时构造对应的证据集合，包括：

- 对应规范来源。
- 条款或片段内容。
- 页码信息。
- 原文在 PDF 中的位置。

从实现上看，证据不是完全交给模型自由生成，而是大量依赖检索阶段保留下来的结构化元数据来构造。这一点能明显提升出处展示的稳定性。

### 第七步：前端展示答案与证据

前端拿到结果后，一边展示中文回答，一边展示引用来源。用户点击证据后，可以直接打开对应 PDF 页面，系统会尽量进行高亮定位；如果精确位置不足，也会至少定位到对应页面。

这让系统的最终使用体验更接近“智能查标工具”，而不是普通聊天窗口。

## 5. 系统用了哪些关键技术或框架

下面这些技术不是泛泛而谈，而是从当前实现中可以明确看出的关键能力。

### 5.1 后端接口与流式响应

系统后端采用 Python Web API 框架来承载问答服务，并支持流式输出。

这意味着用户可以看到答案逐步生成，而不是必须等到整段结果全部完成后再返回。对于问答场景，这种体验更接近真实助手，也有利于后续扩展成长回答和多轮交互。

### 5.2 大模型接入方式

系统通过 OpenAI 兼容接口接入大模型服务，而不是把模型逻辑写死在某一家平台上。

这带来两个好处：

- 模型替换成本较低，后续可以灵活切换。
- 便于在不同模型之间做成本、速度和效果的平衡。

根据代码推测，当前默认配置更偏向接入 DeepSeek 一类的兼容模型，同时保留了请求级模型切换能力。

### 5.3 文档解析能力

系统使用专门的 PDF 解析方案，把规范文档转成 Markdown 和结构化元数据，并保留页面位置、内容列表等信息。

这一步不是简单 OCR，而是为了支持后续的章节结构理解、条款定位、表格和公式处理，因此是整个技术路线的底座。

### 5.4 向量检索与关键词检索结合

系统采用了混合检索路线：

- 向量检索负责理解语义相近的问题。
- 关键词检索负责命中明确术语、章节编号、公式编号和表格编号。

在基础设施上，系统分别使用向量数据库和全文检索引擎来承担这两种任务，然后在应用层做结果融合。

这种做法比“只用向量库”或“只用关键词搜索”更适合规范问答场景。

### 5.5 Embedding 与重排模型

系统在检索链路中使用了专门的向量表示模型和重排模型。

它们的作用分别是：

- 把文本转换成向量表示，支持语义检索。
- 在初步召回之后，再对结果进行相关性重排，提高前几条结果的准确度。

从当前实现看，这部分采用了 BGE 系列模型路线。这是目前中文和多语言检索场景里较常见、也较成熟的一类方案。

### 5.6 前端技术与证据展示

前端采用现代 Web 单页应用技术栈，支持问答会话、流式展示、Markdown 渲染、公式展示和 PDF 查看。

其中，PDF 展示并不是简单嵌入，而是和后端返回的定位信息联动，尽量把用户带到对应条款所在的位置。这对于“答案可核查”这个目标非常关键。

### 5.7 会话状态与结果持久化

系统内部保留了短期会话记忆，同时前端也会把会话状态做本地持久化。

这说明系统已经不只是一次性问答 Demo，而是在向真实工作流工具靠拢：用户刷新页面或短时离开后，仍能回到原来的问答上下文。

## 6. 这条技术路线为什么适合这个场景

工程规范问答和普通知识问答有明显不同，这也是这条路线成立的根本原因。

### 6.1 规范问答需要“依据”，不是只要“像样的回答”

在工程场景里，一个回答是否有用，不仅取决于说得通不通，更取决于能不能回到具体条款。

因此，这个系统不是把大模型当成“万能专家”，而是把它放在“检索结果之后”，让它围绕已找到的规范内容做解释。这比纯聊天机器人更可靠。

### 6.2 规范文本既有语义问题，也有编号问题

用户有时会问概念性问题，比如某种设计条件如何理解；有时会问编号性问题，比如某个表格、公式或条款在哪里。

所以系统采用混合检索非常合理：

- 语义检索解决“意思接近但说法不同”的问题。
- 关键词检索解决“编号、术语、固定表达”的问题。

这比单一路线更贴近规范文档的实际使用方式。

### 6.3 中问英标是一个天然的跨语言检索场景

这个场景最大的难点之一，就是用户用中文表达，而规范主要是英文。

系统先做中文问题理解，再做英文检索扩展和术语映射，这种路线明显比“直接拿中文去搜英文文档”更适合实际业务。

### 6.4 工程文档中表格和公式占比高

规范不只是自然语言文本，还包含大量表格、公式和编号结构。

系统在离线阶段就对这些内容做了单独处理，并在检索阶段尽量把它们纳入召回范围。这样更符合工程规范的真实文档形态。

### 6.5 最终交付形态接近“查标助手”

用户最终看到的不是单纯聊天记录，而是：

- 中文回答。
- 条款来源。
- 页码信息。
- PDF 原文定位。

这种交付形态更适合工程设计、审查和沟通环节，也更容易进入实际业务流程。

## 7. 当前系统大概处于什么阶段

根据代码推测，这个系统已经超过“纯原型”阶段，但还没有完全达到成熟商业产品阶段，更接近于“可用演示版到初步产品化版本之间”。

这样判断，主要有几个依据：

- 已经具备完整的离线处理、在线问答、前端展示和证据回显闭环。
- 已经具备流式回答、会话管理和问答接口等基础产品能力。
- 已经开始做检索效果评测，而不是只停留在功能实现层面。
- 同时，也能看出部分能力还偏演示性质，系统覆盖范围和产品化细节还有继续完善空间。

如果用于汇报，可以把当前阶段表述为：

“已完成核心技术闭环，具备试用和演示条件，正在从可用系统向稳定产品演进。”

## 8. 后续可以怎么演进和优化

结合当前实现，后续演进方向可以重点放在以下几个方面。

### 8.1 更精准的条款定位

继续提升页码、段落、表格、公式的定位精度，让用户点击证据后更快落到准确位置，而不是只到页面级别。

### 8.2 多文档联合检索

当前架构已经具备多文档能力基础，后续可以进一步强化跨规范、跨章节、跨专业的联合检索，让系统支持更复杂的工程问题。

### 8.3 表格与公式深度理解

后续可以在现有基础上，把表格和公式从“可被检索”进一步提升为“可被结构化理解”，例如：

- 识别公式变量含义。
- 解释表格字段和适用条件。
- 辅助单位换算和参数判断。

这会大幅提升工程规范问答的专业深度。

### 8.4 多轮对话与上下文追踪

目前系统已经有短会话基础，下一步可以增强多轮连续追问能力，例如支持：

- 追问同一条款的适用边界。
- 对前一问结果继续展开。
- 在多个规范之间做连续比较。

### 8.5 更完整的评测体系

后续不应只关注“模型答得像不像”，还应进一步量化：

- 条款召回率。
- 引用准确率。
- 定位准确率。
- 用户核查成本。

这样系统才能从“可演示”走向“可持续优化”。

### 8.6 更强的产品化能力

例如：

- 文档批量导入与更新流程。
- 权限控制和日志留痕。
- 运行监控与异常告警。
- 面向企业内部部署的配置管理。

这些能力会决定系统能否真正进入工程组织内部长期使用。

### 8.7 规范差异对比能力

从业务价值看，后续非常值得扩展“欧标与国标差异解释”能力。

因为用户真正关心的，不只是“欧标怎么写”，还包括“它和中国规范有什么不同、为什么不同、工程上意味着什么”。这将使系统从“查标工具”进一步升级为“规范迁移辅助工具”。

## 面向管理层的技术路线总结

这套系统的核心思路，不是让大模型直接回答工程规范问题，而是先把欧洲规范文档做结构化入库，再通过检索找到相关条款，并基于这些条款生成中文解释，同时把原文依据、页码和定位信息一起展示出来。这样既保留了 AI 在理解和表达上的优势，又降低了“回答看起来合理但缺少依据”的风险。对跨语言、跨规范体系的工程应用场景来说，这是一条更稳妥、也更容易落地的技术路线。
